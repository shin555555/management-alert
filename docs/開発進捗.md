#### 🔴 Critical（必ず修正）

**[C-1] overdue/red の分類不整合 — ユーザーに誤情報を与えるバグ**

- **問題**: `actions.ts` の `determineAlertLevel()` は期限超過タスクを `alertLevel = "red"` として返す。しかし `groupTasksByMonth()` 内（行 131）では `isOverdue = new Date(t.endDate) < now && !t.alertLevel` という条件で超過を判定しており、`alertLevel` が `"red"` の場合は `!t.alertLevel` が `false` になるため `summary.overdue` にカウントされない。結果として **グループヘッダーに「超過 0」と表示されつつ、行には「緊急」バッジが表示される**という矛盾が生じる。
- **影響**: スタッフが「超過タスクはない」と誤認する可能性がある。
- **修正方法**:
  ```ts
  // dashboard-view.tsx の groupTasksByMonth 内、summary集計ループを以下に修正
  for (const t of monthTasks) {
    const statusFlow = t.statusFlow;
    const firstStatus = statusFlow[0];
    const isWaiting = t.currentStatus === firstStatus;
    const isOverdue = new Date(t.endDate) < now; // ← !t.alertLevel を削除

    if (isOverdue) summary.overdue++;            // ← 最優先でカウント（overdue は red より上位）
    else if (t.alertLevel === "red") summary.red++;
    else if (t.alertLevel === "orange") summary.orange++;
    else if (t.alertLevel === "yellow") summary.yellow++;
    else if (isWaiting) summary.waiting++;
    else summary.inProgress++;
  }
  ```
  また、`TaskRow` のバッジ表示（行 440–456 付近）でも `isOverdue` を優先チェックし「超過」バッジを表示するよう調整する。

---

**[C-2] useState の初期値がデータ更新後に再計算されない**

- **問題**: `useState(defaultOpen)` の初期値は**初回マウント時にしか適用されない**。`router.refresh()` でタスクが更新され `monthGroups` が変わっても `openGroups` は古い状態のまま。「直近2ヶ月はデフォルト展開」という要件が更新後に破綻する。
- **影響**: ステータス更新後に展開状態がずれる（新しい月グループが展開されないなど）。
- **修正方法**:
  ```ts
  // useMemo で monthGroups を計算した後、useEffect で openGroups を同期する
  const monthGroups = useMemo(() => groupTasksByMonth(tasks), [tasks]);

  const [openGroups, setOpenGroups] = useState<Set<string>>(
    new Set(monthGroups.slice(0, 2).map((g) => g.key))
  );

  // tasks が変わったら openGroups をリセット
  useEffect(() => {
    setOpenGroups(new Set(monthGroups.slice(0, 2).map((g) => g.key)));
  }, [monthGroups]);  // monthGroups が変わった（キーセットが変わった）ときだけ実行
  ```

#### 🟠 Important（修正推奨）

**[I-1] 和暦ラベル生成の `.replace(/1日$/, "")` が脆弱**

- **場所**: `dashboard-view.tsx` 行 115 付近
- **問題**: `formatToWareki(new Date(year, month - 1, 1)).replace(/1日$/, "")` は `formatToWareki` の出力フォーマットが変わると壊れる。
- **修正方法**: `wareki.ts` に月のみのフォーマッターがないため、ここは直接計算で問題ない:
  ```ts
  const reiwaYear = year - 2018;
  const label = `令和${reiwaYear}年${month}月`;
  ```

---

**[I-2] `isOverdue` の日付比較にタイムゾーン正規化が不足**

- **場所**: `dashboard-view.tsx` 行 389–390 付近（`TaskRow` 内の `isOverdue` 判定）
- **問題**: `new Date(task.endDate) < new Date()` はタイムゾーン差で最大±1日ずれる可能性がある。サーバー側（`determineAlertLevel`）でも同様の処理があり、現在は偶然一致しているが潜在的なバグ。
- **修正方法**: `date-fns` の `startOfDay` を使って両辺を日付の開始に正規化する:
  ```ts
  import { startOfDay } from "date-fns";
  const isOverdue = startOfDay(new Date(task.endDate)) < startOfDay(new Date());
  ```

---

**[I-3] `<button>` に `type="button"` がない**

- **場所**: `MonthSection` コンポーネントの折りたたみボタン（行 282 付近）
- **問題**: `type` 属性なしの `<button>` はフォーム内に入った場合 `type="submit"` として動作し、意図しないフォーム送信が起こりえる。
- **修正方法**: `<button type="button" ...>` に変更する。「すべて展開」「すべて折りたたみ」ボタンも同様に確認・修正。

---

**[I-4] アクセシビリティ: 折りたたみボタンに `aria-expanded` がない**

- **場所**: `MonthSection` の折りたたみボタン（行 280–334 付近）
- **問題**: スクリーンリーダーが展開・折りたたみ状態を読み上げられない。
- **修正方法**:
  ```tsx
  <button type="button" aria-expanded={isOpen} ...>
  ```
  必要に応じて `aria-controls="month-section-{key}"` とターゲット div の `id={`month-section-${key}`}` も追加する。

---

#### 🟡 Minor（任意）

**[M-1] `MonthSection` の Props 型を `interface` で定義する**

- コーディングスタイル規約「Props 型はコンポーネントファイル内で `interface SomeProps {}` として定義」に従い、インライン匿名型から named interface に変更。

**[M-2] `expandAll`/`collapseAll` を `useCallback` でメモ化**

- ユーザーインタラクション時のみ呼ばれる関数なので優先度は低い。
