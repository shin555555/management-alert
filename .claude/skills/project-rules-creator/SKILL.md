# Claude Rules Creator
 
プロジェクトの `.claude/rules` ディレクトリに最適化されたClaudeルールを作成・管理するスキル。
 
---
 
## このスキルの目的
 
`.claude/rules/` は、Claude（AIアシスタント）がプロジェクト固有のルール・規約・ガイドラインを参照するためのディレクトリ。このスキルは：
 
1. プロジェクトを分析し、必要なルールを特定する
2. 既存ルールを読み込み、改善・統合する
3. 最適化された Markdown ルールファイルを生成する
 
---
 
## ステップ1: 既存ルールの読み込み
 
まず、既存の `.claude/rules` ディレクトリを確認する。
 
```bash
# プロジェクトのルートを特定
ls -la .claude/rules/ 2>/dev/null || echo "NO_RULES_DIR"
```
 
- ディレクトリが存在する場合 → 全ファイルを読み込む（`view` ツールを使用）
- 存在しない場合 → 新規作成フローへ進む
 
既存ファイルがある場合、以下を把握する：
- 既存ルールの構造・カバー範囲
- 欠けている領域
- 改善できる部分
 
---
 
## ステップ2: プロジェクト分析
 
プロジェクトの性質を理解するため、以下を調査する：
 
```bash
# プロジェクト構造の確認
ls -la
cat package.json 2>/dev/null || cat pyproject.toml 2>/dev/null || cat Cargo.toml 2>/dev/null || echo "No package file"
cat README.md 2>/dev/null | head -50
```
 
収集すべき情報：
- **技術スタック**: 言語、フレームワーク、主要ライブラリ
- **プロジェクト種別**: Web app / CLI / ライブラリ / マイクロサービス など
- **チーム規模・スタイル**: 個人 / チーム開発
- **既存の設定ファイル**: `.eslintrc`, `.prettierrc`, `tsconfig.json` など
 
---
 
## ステップ3: ユーザーへのヒアリング
 
以下の質問でルールの方向性を確認（すべてを一度に聞く）：
 
### 必須確認事項
1. **コーディングスタイル**: 命名規則、コメント言語（日本語/英語）、インデントなど
2. **アーキテクチャ原則**: ディレクトリ構造、モジュール分割方針、禁止パターン
3. **出力フォーマット**: Claudeに期待する回答スタイル（詳細/簡潔、コードのみ/説明付きなど）
4. **禁止事項**: 使ってはいけないライブラリ、パターン、アプローチ
5. **特殊要件**: セキュリティ要件、パフォーマンス基準、アクセシビリティなど

## ステップ4: ルールファイルの設計
 
### ファイル構成パターン
 
**シンプル（小規模プロジェクト）**:
```
.claude/rules/
└── main.md          # 全ルールを1ファイルに集約
```
 
**標準（中規模プロジェクト）**:
```
.claude/rules/
├── coding-style.md   # コーディング規約
├── architecture.md   # アーキテクチャ方針
├── workflow.md       # 開発ワークフロー
└── ai-behavior.md    # Claudeへの行動指示
```
 
**詳細（大規模・チーム開発）**:
```
.claude/rules/
├── 00-overview.md          # 全体概要・インデックス
├── 01-coding-style.md      # コーディングスタイル
├── 02-architecture.md      # アーキテクチャ
├── 03-testing.md           # テスト方針
├── 04-security.md          # セキュリティ
├── 05-ai-behavior.md       # Claudeへの指示
└── domain/
    ├── frontend.md          # フロントエンド固有
    └── backend.md           # バックエンド固有
```
 
プロジェクト規模に応じて最適な構成を選択する。

## ステップ5: ルールファイルの記述原則
 
### 効果的なルールの書き方
 
**✅ 良い例**:
```markdown
## 命名規則
- 変数・関数: camelCase（例: `getUserData`, `isLoading`）
- コンポーネント: PascalCase（例: `UserCard`, `LoginForm`）
- 定数: SCREAMING_SNAKE_CASE（例: `MAX_RETRY_COUNT`）
- ファイル: kebab-case（例: `user-profile.tsx`）
```
 
**❌ 悪い例**:
```markdown
## 命名規則
適切な命名規則を使うこと。
```
 
### ルール記述の要素
各ルールには以下を含める：
1. **What（何を）**: 具体的なルール内容
2. **Why（なぜ）**: 理由・背景（任意だが推奨）
3. **Example（例）**: コード例や具体例
4. **Exception（例外）**: ルールの例外条件（あれば）
 
### Claudeへの行動指示テンプレート
 
```markdown
## Claudeへの指示
 
### 回答スタイル
- コードの変更は差分（diff形式）ではなく、完全なファイル内容で提示する
- 説明は日本語で行い、コードとコメントは英語で書く
- 実装前に設計・方針を確認してから進める
 
### 禁止事項
- `any` 型の使用（TypeScript）
- console.log をコミットに含めない
- 直接 DOM 操作（React プロジェクト）
 
### 推奨アプローチ
- エラーハンドリングは必ず実装する
- 関数は単一責任の原則に従う
- テストを書いてから実装する（TDD）
```

```bash

mkdir -p .claude/rules
```
 
各ファイルを `create_file` ツールで生成する。
 
### 生成後の確認
```bash
# 生成されたファイルを確認
ls -la .claude/rules/
```

## ステップ7: ルールの検証・改善
 
生成したルールをユーザーに確認してもらい：
 
1. **完全性チェック**: 漏れているルールはないか？
2. **明確性チェック**: 曖昧な表現はないか？
3. **実用性チェック**: 実際のコーディングで使えるか？
 
フィードバックをもとに改善を繰り返す。

## 重要な注意事項
 
- ルールは「制約」ではなく「チームの共通認識」として書く
- 過度に細かいルールは逆効果 → 本当に重要なものに絞る
- 定期的な見直しを促す（プロジェクト進行に合わせて更新）
- `.claude/rules/` の存在を README.md に記載することを推奨
