# Doc-Sync スキル
 
## 概要
 
このスキルは、**実装済みコード（Ground Truth）** と **プロジェクトドキュメント（タスクリスト・進捗記録・仕様書など）** の乖離を自動検出し、ドキュメントを現実に合わせて更新するワークフローです。
 
どんなプログラミング言語・フレームワーク・プロジェクト構造にも対応します。
 
---
 
## 実行ワークフロー
 
### Step 1: ドキュメントファイルの自動検出
 
プロジェクトルートから以下のパターンでドキュメントを検索・読み込む：
 
**優先度高（必ず読む）:**
- `task.md`, `tasks.md`, `TODO.md`, `todo.md`
- `PROGRESS.md`, `progress.md`, `開発進捗.md`, `進捗.md`
- `implementation_plan.md`, `IMPLEMENTATION.md`, `実装計画.md`
- `ROADMAP.md`, `roadmap.md`
- `CHANGELOG.md`（未記載の変更があるか確認用）
 
**優先度中（存在すれば読む）:**
- `docs/` ディレクトリ配下の `.md` ファイル
- `README.md`（実装状況の記述がある場合）
- `sprint.md`, `backlog.md`, `issues.md`
 
**検索コマンド例:**
```bash
find . -maxdepth 3 -name "*.md" | grep -iE "(task|todo|progress|進捗|plan|roadmap|backlog|sprint|実装)" | head -20
```
 
未チェック（`[ ]`）のタスク項目を全て抽出してリストアップする。
 
---
 
### Step 2: 実装ファイルの確認
 
未チェック項目それぞれについて、対応する実装ファイルが実際に存在・機能しているか検証する。
 
**確認方法（言語・プロジェクト種別に応じて選択）:**
 
| 確認対象 | 確認方法 |
|----------|----------|
| ファイル存在 | `ls`, `find` でパス確認 |
| 関数・クラス実装 | `grep -n "def \|class \|function \|const \|export"` |
| APIエンドポイント | ルーティングファイルを `grep` |
| UIコンポーネント | コンポーネントファイルの存在確認 |
| テスト通過 | テスト結果ファイル・CIログ確認（存在する場合） |
| DB定義 | マイグレーションファイルの存在確認 |
| 設定・環境変数 | 設定ファイルのキー確認 |
 
**判定基準:**
- ✅ **実装済み**: ファイルが存在し、対象の関数/コンポーネント/ルートが確認できる
- ⚠️ **部分実装**: ファイルは存在するが、該当機能が見当たらない or スタブ状態
- ❌ **未実装**: 関連ファイルが存在しない
 
---
 
### Step 3: チェックボックスの自動更新
 
Step 2 で **実装済み** と判定されたタスクのチェックボックスを更新する。
 
**更新ルール:**
- `[ ]` → `[x]` に変更（完了確認できたもの）
- `[ ]` のまま維持（未実装・部分実装のもの）
- 既存の `[x]` は絶対に変更しない
 
**更新時の注意:**
- 1ファイルずつ確実に更新する（まとめて書き換えない）
- 更新前後の diff を内部で確認する
- 曖昧な場合はユーザーに確認してから更新
 
---
 
### Step 4: 追記漏れの補完
 
ドキュメントに記録されていない実装上の決定事項を補完する：
 
**補完対象の例:**
- 新たに追加されたファイル・モジュールの説明
- 実装中に変更されたアーキテクチャの決定
- 追加された依存ライブラリ・ツール
- 発見・対処したバグとその解決策
- 次回セッションへの申し送り事項
 
補完先は内容に応じて適切なドキュメントを選ぶ（進捗記録 / 実装計画 / README など）。
 
---
 
### Step 5: 同期完了レポートの出力
 
ユーザーへの報告は以下の形式で行う：
 
```
## ドキュメント同期完了 ✅
 
### 更新したチェックボックス
- [x] ○○機能の実装（task.md, 行XX）
- [x] △△APIエンドポイントの追加（task.md, 行XX）
 
### 変更なし（実装未確認のため保留）
- [ ] □□機能（対応ファイルが見つからず）
 
### 追記した内容
- 開発進捗.md: △△の実装方針を追記
- README.md: セットアップ手順を更新
 
### 次回セッションへの申し送り
- ◇◇機能が途中のため、○○ファイルから再開予定
```

## エッジケース対応
 
### ドキュメントが見つからない場合
- ユーザーにドキュメントのパスを確認する
- または `find . -name "*.md"` で全 .md ファイルを列挙してユーザーに選択を促す
 
### チェックボックスの確信が持てない場合
- 部分実装の場合は `[-]`（進行中）を使うか、コメントを追記するかをユーザーに確認
- 判断が難しい項目はリストアップしてユーザーに委ねる
 
### 大規模プロジェクトの場合
- タスク数が多い場合は、セクション単位で区切って段階的に処理
- 優先度の高いタスク（直近の sprint / マイルストーン）から処理
 
### ドキュメントが複数言語混在の場合（日本語+英語など）
- 既存の言語スタイルを維持して追記する
- 新規追記は既存ドキュメントの言語に合わせる

## 使用例
 
**典型的なトリガー:**
- 「セッション終わりにdoc同期お願い」
- 「task.mdを最新の実装に合わせて更新して」
- 「実装したけどドキュメントが古いままなので整理して」
- `/sync`
- 「次回セッションのために進捗を整理しておいて」
 
**このスキルが解決する問題:**
1. 「実装したのに `task.md` のチェックを入れ忘れた」
2. 「どこまで実装したか次回セッションで分からなくなる」
3. 「ドキュメントと実際のコードが乖離していて混乱する」
4. 「/onboard コマンドで古い情報を読み込んでしまう」
